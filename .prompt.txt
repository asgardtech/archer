# Issue #20: Make this project a collection of games, not just Archer

**Original issue:** Make it so this project is actually a collection of games, not just archer; archer should be the first game in the list.

## Technical Specification

### 1. Summary

The project is currently a single-game application called "Archer" (a balloon-shooting Canvas game). This change transforms the repository into a **game collection platform** with a launcher/menu screen where players can browse and select from a list of games. The existing Archer game becomes the first entry in that collection. The architecture must make it trivial to add new games in the future by dropping a module into a `games/` directory and registering it.

### 2. Architecture & Design Decisions

#### 2.1 Game Interface Contract

Every game in the collection must implement a common `IGame` interface so the launcher can manage them uniformly:

```typescript
export interface GameDescriptor {
  id: string;
  name: string;
  description: string;
  thumbnailColor: string; // fallback color for card background
  createGame: (canvas: HTMLCanvasElement) => IGame;
}

export interface IGame {
  start(): void;
  stop(): void;
  destroy(): void; // tear down event listeners, timers, etc.
  onExit: (() => void) | null; // callback to return to the launcher
}
```

- `createGame` is a factory function (not a constructor reference) so games can encapsulate their own setup.
- `onExit` is set by the launcher so each game can signal "return to menu" without knowing about the launcher.
- `destroy()` ensures clean teardown of input listeners and animation frames before the launcher regains control.

#### 2.2 Launcher

A new `Launcher` class replaces the current root-level entry point logic. It:
- Renders a title screen and a scrollable/grid list of game cards on the shared canvas.
- Handles mouse/touch input to detect which game card was clicked.
- On selection, creates the chosen game via its factory, passes it the canvas, sets `onExit`, and calls `start()`.
- When a game fires `onExit`, the launcher calls `game.destroy()` and re-renders itself.

The launcher draws directly on the same `<canvas>` element — no DOM-based menus — keeping the project consistent with its current Canvas-only rendering approach.

#### 2.4 Changes to Archer (`Game.ts` → `ArcherGame.ts`)

The existing `Game` class is renamed to `ArcherGame` and adapted:

1. **Implements `IGame`**: adds `destroy()` and `onExit` members.
2. **`destroy()`**: cancels the animation frame, removes event listeners registered by `InputManager`, removes the resize/orientation-change listeners.
3. **Menu → Exit behavior**: When the game is in `"menu"` state (after game-over or victory and the player clicks), instead of restarting, it calls `this.onExit?.()` to return to the launcher. The Archer's internal `"menu"` state is replaced by `"title"` (pre-game splash) which transitions to `"playing"` on first click, while post-game clicks invoke `onExit`.
4. **`InputManager` cleanup**: Add a `destroy()` method that calls `removeEventListener` for all registered handlers (store references to bound handlers).

All existing game logic (levels, entities, systems, rendering) stays **unchanged** beyond the file move and the adapter layer above.

### 5. Data Models / Contracts

#### 5.1 `GameDescriptor` (registration metadata)

```typescript
interface GameDescriptor {
  id: string;           // unique slug, e.g. "archer"
  name: string;         // display name, e.g. "Balloon Archer"
  description: string;  // short tagline
  thumbnailColor: string; // CSS color for card background
  createGame: (canvas: HTMLCanvasElement) => IGame;
}
```

#### 5.2 `IGame` (runtime contract)

```typescript
interface IGame {
  start(): void;        // begin the game loop
  stop(): void;         // pause (keep state)
  destroy(): void;      // full teardown: cancel rAF, remove listeners
  onExit: (() => void) | null;  // launcher sets this; game calls it to return
}
```

No network APIs, databases, or server-side contracts are involved — this is a purely client-side Canvas application.

### 6. Launcher UX Details

- **Title**: Large heading at the top (e.g., "Game Collection" or a brand name).
- **Game cards**: Rendered as rounded rectangles in a grid/row. Each card shows the game's `name`, `description`, and uses `thumbnailColor` as a background fill. Hover state indicated by border highlight.
- **Selection**: Clicking a card transitions to that game. The canvas is cleared and handed to the game's factory.
- **Return flow**: When `onExit` fires, the launcher re-renders. No page reload.
- **Single game**: When only one game exists, the launcher still shows the grid (with one card). This keeps the pattern consistent and avoids special-case logic.
- **Touch support**: The launcher reuses the same canvas coordinate mapping pattern as Archer's `InputManager`.

### 7. Edge Cases & Error Handling

| Edge case | Handling |
|---|---|
| Game's `onExit` called multiple times | Guard with a `destroyed` flag; only process the first call. |
| `destroy()` called while game loop is mid-frame | Set `running = false` before calling `cancelAnimationFrame`. The loop's guard clause (`if (!this.running) return`) prevents further updates. |
| Canvas resize while launcher is active | Launcher registers its own resize handler; tears it down when handing off to a game. |
| Empty game registry | Launcher renders a "No games available" message instead of crashing. |
| Browser back button / navigation | Not applicable — single-page Canvas app with no routing. |
| `createGame` throws | Launcher catches the error, logs it, and stays on the menu. Displays an error toast on the canvas. |
| Game added without registering | Won't appear — only registered games in `GAME_REGISTRY` are shown. This is by design (explicit > implicit). |
| Event listener leaks on game switch | `InputManager.destroy()` removes all listeners. Launcher verifies cleanup by owning the lifecycle. |
| Multiple rapid game selections | Launcher disables input briefly during transition (debounce). |
| Window loses focus during game | Existing `DT_CAP` in Archer prevents time jumps. Each game is responsible for its own focus handling. |

### 8. Migration Strategy

This is a **non-breaking refactor** for the Archer game itself — all game logic, levels, entities, and systems are unchanged. The steps are:

1. Create `src/shared/types.ts` with `IGame` and `GameDescriptor`.

4. Adapt `ArcherGame` to implement `IGame` (add `destroy()`, `onExit`, modify menu-state behavior).

6. Create `src/games/archer/index.ts` exporting the descriptor.

8. Rewrite `src/index.ts` to use the launcher.
